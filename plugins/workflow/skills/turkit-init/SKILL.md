# Turkit Init

Scan the current repo and propose a `.turkit.yaml` matching its build tool, package manager, base branch, issue tracker, workflow, and output overrides. Personal preferences can be inherited from global Turkit config. The operator reviews and confirms before the file is written.

## Steps

1. **Refuse to overwrite silently.** If `.turkit.yaml` already exists, read it first, show a diff of the proposed change, and ask whether to merge / overwrite / skip. Never overwrite without consent.

2. **Read global preferences if present.** Check `~/.config/turkit/config.yaml`, then `~/.turkit.yaml`. Treat these as inherited personal preferences, not project facts. Only inherit `output.*` and `workflow.token_budget`; ignore global project-specific keys such as `commands`, `base_branch`, `rules`, `preview`, and `vcs`.

3. **Detect the build tool / package manager.** Scan the repo root in this order:

   - `package.json` + `pnpm-lock.yaml` → pnpm
   - `package.json` + `bun.lockb` → bun
   - `package.json` + `yarn.lock` → yarn
   - `package.json` alone → npm
   - `Justfile` → just
   - `Makefile` → make
   - `Cargo.toml` → cargo
   - `pyproject.toml` + `poetry.lock` → poetry
   - `pyproject.toml` + `uv.lock` → uv
   - `go.mod` → go
   - Nothing recognizable → leave `commands:` empty; the operator fills it in manually.

4. **Resolve commands.** For the detected tool, map to turkit's commands: `dev`, `check`, `lint`, `fmt`, `test`, `build`.

   - **package.json (pnpm / bun / yarn / npm)** — read `scripts.*` and pick by name convention:
     - `dev`: prefer `dev`, then `serve`.
     - `check`: prefer `typecheck`, then `check`, then `tsc --noEmit` (only if `tsconfig.json` exists).
     - `lint`: prefer `lint`, then `eslint .` (only if `.eslintrc*` or `eslint.config.*` exists).
     - `fmt`: prefer `format`, `fmt`, then `prettier --write .` (only if `.prettierrc*` or `prettier.config.*` exists).
     - `test`: prefer `test`, `vitest`, `jest`.
     - `build`: prefer `build`, `vite build`, `tsc --build`.
   - **Justfile / Makefile** — look for targets named `dev`, `check`, `lint`, `fmt`, `test`, `build`. Use `just <target>` / `make <target>` verbatim.
   - **cargo** — `cargo check`, `cargo clippy`, `cargo fmt`, `cargo test`, `cargo build`.
   - **poetry / uv / go** — canonical commands per `references/build-tool-detection.md`.
   - **dev command** — never invent a `dev` command for non-JavaScript ecosystems. Only include it when the repo exposes an explicit script/target.

   Omit any command you cannot resolve with confidence. Do not invent scripts. The runtime fallback in the build-tool contract will pick up the rest.

5. **Detect the base branch.** In order:
   - `git symbolic-ref refs/remotes/origin/HEAD 2>/dev/null` (strip `refs/remotes/origin/`).
   - Fallback: `main`. Never default to `master` — if the repo uses `master` you'll detect that via the symbolic-ref.

6. **Detect the issue tracker.** Report findings per `references/issue-tracker-detection.md`:
   - MCP match (name + how to invoke)
   - Branch-name pattern only (show the current branch + extracted ticket ID if any)
   - Nothing
   This is informational — no config key is written for tracker detection.

7. **Propose the config.** Emit the full `.turkit.yaml` in a fenced block with a top comment that summarizes detection and inherited global preferences:

   ```yaml
   # Generated by /turkit:turkit-init — detected <tool>, base <branch>, tracker <summary>
   commands:
     <key>: <resolved command>
     ...
   base_branch: <branch>
   workflow:
     token_budget: low # include only when not inherited globally, or when this repo overrides it
   output:
     language: en # include only when this repo overrides the global/default output preference
   ```

   Omit `output` and `workflow.token_budget` when global config already covers them and the repo does not need an override. Mention the inherited global values in the summary instead of copying them into the repo.

8. **Seed the quality baseline.** If the repo has no rules doc (no `CLAUDE.md` / `AGENTS.md` with coding rules, no `docs/conventions/*.md`), propose writing `docs/conventions/code-quality.md` from [`references/rules-baseline.md`](references/rules-baseline.md) and adding to the proposed config:

   ```yaml
   rules:
     docs:
       - docs/conventions/code-quality.md
   ```

   The project owns the copy afterwards — sharpening, additions, and documented tradeoffs live in the repo, not in the baseline. If rules docs already exist, skip seeding and recommend `rules-refresh` instead.

9. **Ask before writing.** Show the proposed config + a one-line summary of what was detected, inherited, and omitted. Then: *"Write `.turkit.yaml` (and `docs/conventions/code-quality.md` when proposed) at the repo root? [y/n]"*. Only write on explicit yes.

## Guardrails

- Never overwrite `.turkit.yaml` without showing the diff and getting consent.
- Seeding only creates new files — never overwrite an existing rules doc.
- Never invent commands that won't actually run (don't default to `tsc --noEmit` without `tsconfig.json`, don't default to `prettier` without a prettier config, etc.).
- Never add a `tracker:` key or similar — tracker detection is fully runtime, not config-driven.
- Apply `references/output-preferences.md` for operator-facing language/style.

## Example — pnpm / TypeScript project

Typical detection on a Vite + TypeScript repo with pnpm:

```yaml
# Generated by /turkit:turkit-init — detected pnpm, base main, Linear MCP available
commands:
  dev: pnpm dev
  check: pnpm typecheck
  lint: pnpm lint
  fmt: pnpm format
  test: pnpm test
  build: pnpm build
base_branch: main
workflow:
  token_budget: low
output:
  style: compact
  language: auto
  technical_terms: keep-english
```
